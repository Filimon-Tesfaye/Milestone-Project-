Core Ideas Behind HTTP

HTTP is an application‑level protocol that follows a request–response model. A client sends a request message, and a server replies with a response message. The details of those messages are defined by the HTTP specification.

HTTP Request Structure

An HTTP request typically includes:

  • A request line – method, target URL, and protocol version, for example: GET /index.html HTTP/1.1.
  • Headers – key‑value pairs that carry metadata such as Host, User-Agent, and Accept.
  • An optional message body – used with methods such as POST when sending form data or JSON.

Common HTTP Methods

  • GET – retrieve a representation of a resource.
  • POST – submit data to be processed (for example, form data).
  • PUT – replace the representation of a resource.
  • DELETE – request that a resource be removed.
  • HEAD – same as GET but returns only headers.

Status Codes

Status codes indicate the result of a request. They are grouped into ranges:

  • 1xx – Informational (rarely seen directly in browsers).
  • 2xx – Success (for example, 200 OK).
  • 3xx – Redirection (for example, 301 Moved Permanently).
  • 4xx – Client errors (for example, 404 Not Found).
  • 5xx – Server errors (for example, 500 Internal Server Error).

Headers and Content Negotiation

HTTP headers allow clients and servers to exchange metadata. For example, the Accept header lets a client specify which content types it prefers (such as text/html or application/json), and the Content-Type header tells the client how to interpret the response body.

State and Cookies

HTTP itself is a stateless protocol, which means each request is handled independently. To maintain user sessions (such as staying logged in), websites commonly use cookies. Cookies are small pieces of data that a server can set in the browser and that the browser sends back with later requests to the same site.
